1 Overview

This document describes the API test suite for the MU Admin App — the Express/Node.js/MongoDB management system used to tag Maynooth University modules with Data, Digital, and Critical Skills classifications.

Tests use Jest as the test runner and Supertest to make HTTP requests directly to the Express app without starting a real server. All tests run against a live MongoDB Atlas connection to ensure real integration coverage — no mocking.

All 35 tests pass. Run time approximately 18 seconds. Tests cover authentication, authorisation, role-based access control, export data integrity, JWT tampering, NoSQL injection, and privilege escalation.

Change to server.js

One small change was made to server.js to make it testable. The app.listen() call was wrapped in a require.main === module guard, and module.exports = app was added at the end: 

// Before — server always started on import:
app.listen(PORT, ...);

// After — only starts when run directly (not when imported by tests):
if (require.main === module) {
  app.listen(PORT, ...);
}
module.exports = app;

This is the standard pattern for making an Express app testable with Supertest. It has no effect on normal operation — node server.js behaves identically.

7 Security Tests — security.test.js

Tests that the application correctly defends against common API security threats: JWT tampering, NoSQL injection, and privilege escalation. All 16 security tests pass, confirming the defences in place.

JWT Security (6 tests)

Verifies that the authentication middleware cannot be bypassed by crafting or manipulating tokens. All six cases must return a non-200 status:

GET /api/modules — JWT attacks 6 / 6 passing
Token signed with a different secret → 400
jwt.sign(payload, 'wrong-secret') — server rejects unknown signature
400
Forged token claiming admin role → 400
Payload claims role: 'admin' but signed with wrong secret — rejected
400
Expired token → 400
Token signed with expiresIn: '1ms', used after 50ms delay
400
Token without Bearer prefix → 401
Authorization header set to raw token string — middleware cannot extract it
401
Empty Bearer token → 401
Authorization: Bearer — token part is empty string
401
Structurally valid but fabricated token → 400
Well-formed JWT format but fake signature — rejected by jwt.verify()
400
How JWT verification works: jwt.verify(token, JWT_SECRET) recomputes the token signature using the server's secret. If it doesn't match — whether the token is expired, tampered, or signed with a different key — it throws and the middleware returns 400. The server never reads the payload of an unverified token, so a forged role: 'admin' claim has no effect.

NoSQL Injection (4 tests)

The login route passes the username value directly to MongoDB's findOne(). If a MongoDB query operator (e.g. { $gt: "" }) is passed instead of a string, it could match any document. These tests confirm that no such injection results in a successful login.

POST /api/auth/login — injection attempts 4 / 4 passing
Operator object as username → not 200
Body: { "username": { "$gt": "" }, "password": "anything" }
The injection matches a real user but fails bcrypt password verification
≠ 200
Operator object as password → not 200
Body: { "username": "admin", "password": { "$gt": "" } }
bcrypt.compare() rejects a non-string input
≠ 200
Operator objects in both fields → not 200
Both username and password supplied as { "$gt": "" }
≠ 200
$where operator in username → not 200
Body: { "username": { "$where": "sleep(1000)" } }
≠ 200
Finding worth noting: The injection tests pass (login is not bypassed), but this is partly due to bcrypt's password check acting as a second line of defence. The username field is not sanitised before being passed to MongoDB's findOne(). Adding input sanitisation (e.g. rejecting non-string values) would make the defence explicit rather than incidental.

8 Running the Tests

Run all API tests

cd mumodulesapp
npm run test:api

Run a single file

npx jest tests/api/auth.test.js --forceExit
npx jest tests/api/modules.test.js --forceExit

Watch mode (re-runs on file save)

npx jest --watch --forceExit

Expected output

Test Suites: 3 passed, 3 total
Tests:       35 passed, 35 total
Time:        ~18s
Note: A Force exiting Jest message appears at the end of each run. This is expected — it occurs because the MongoDB Atlas connection opened during the test run is not explicitly closed. The --forceExit flag handles this cleanly. All 19 tests pass before the exit.
