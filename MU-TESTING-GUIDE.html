1 Overview

This guide documents the test suite for the MU Admin App — the Node.js/Express/React management application used to tag Maynooth University modules with Data, Digital, and Critical Skills classifications.

The test suite covers two layers:

  • API tests using Jest and Supertest — fast, headless tests of the Express routes
  • End-to-end (E2E) tests using Playwright — browser-level tests of the full application as a user experiences it
Why both layers? API tests catch broken logic at the server level — fast to run, easy to pinpoint failures. E2E tests catch what API tests can't: UI rendering, browser state, user interaction sequences. Used together they give confidence that a change hasn't broken anything a user would notice.

3 Project Structure

All test files live under a tests/ folder in the mumodulesapp/ directory:

mumodulesapp/
  tests/
    api/
      auth.test.js        ← login endpoint tests
      modules.test.js     ← GET, PUT, export tests
    e2e/
      login.spec.ts       ← login/logout flow
      tagging.spec.ts     ← find module, tag it, save
      filters.spec.ts     ← department/faculty filter
  jest.config.js          ← Jest configuration
  playwright.config.ts    ← Playwright configuration
  .env.test               ← test DB credentials (not committed)
Never commit .env.test — it contains your MongoDB Atlas connection string. Add it to .gitignore alongside .env.

4 Setting Up Jest

Installation

cd mumodulesapp
npm install --save-dev jest supertest

Babel is already configured in the project for Webpack, but Jest needs its own transform config. Add jest.config.js: 

// jest.config.js
module.exports = {
  testEnvironment: 'node',
  testMatch: ['**/tests/api/**/*.test.js'],
  setupFiles: ['dotenv/config'],
};

The setupFiles line loads .env before tests run. For the test database, create .env.test and tell Jest to use it instead: 

// jest.config.js (with test env)
module.exports = {
  testEnvironment: 'node',
  testMatch: ['**/tests/api/**/*.test.js'],
  setupFiles: ['<rootDir>/tests/loadTestEnv.js'],
};
// tests/loadTestEnv.js
const dotenv = require('dotenv');
dotenv.config({ path: '.env.test' });

Add the test script to package.json:

"test:api": "jest --testPathPattern=tests/api",

5 API Tests (Supertest)

Supertest lets you make HTTP requests to your Express app without starting a real server. You import the Express app object and pass it to Supertest — it handles binding to a temporary port automatically.

Key pattern: the Express app (server.js) connects to MongoDB on startup. For tests, the connection string in .env.test points to a separate MU_test database in Atlas, so test writes never touch production data.

Auth tests — tests/api/auth.test.js

const request = require('supertest');
const app     = require('../../server');

describe('POST /api/auth/login', () => {

  it('returns a JWT for valid credentials', async () => {
    const res = await request(app)
      .post('/api/auth/login')
      .send({ username: 'testadmin', password: 'testpass123' });

    expect(res.statusCode).toBe(200);
    expect(res.body).toHaveProperty('token');
  });

  it('returns 401 for wrong password', async () => {
    const res = await request(app)
      .post('/api/auth/login')
      .send({ username: 'testadmin', password: 'wrongpassword' });

    expect(res.statusCode).toBe(401);
  });

  it('returns 400 when fields are missing', async () => {
    const res = await request(app)
      .post('/api/auth/login')
      .send({});

    expect(res.statusCode).toBe(400);
  });

});

Module tests — tests/api/modules.test.js

const request = require('supertest');
const app     = require('../../server');

let token;

// Log in once before all tests to get a JWT
beforeAll(async () => {
  const res = await request(app)
    .post('/api/auth/login')
    .send({ username: 'testadmin', password: 'testpass123' });
  token = res.body.token;
});

describe('GET /api/modules', () => {

  it('returns an array of modules when authenticated', async () => {
    const res = await request(app)
      .get('/api/modules')
      .set('Authorization', `Bearer ${token}`);

    expect(res.statusCode).toBe(200);
    expect(Array.isArray(res.body)).toBe(true);
    expect(res.body.length).toBeGreaterThan(0);
  });

  it('returns 401 without a token', async () => {
    const res = await request(app).get('/api/modules');
    expect(res.statusCode).toBe(401);
  });

});

describe('GET /api/modules/export', () => {

  it('returns an object with totalModules and faculties', async () => {
    const res = await request(app)
      .get('/api/modules/export')
      .set('Authorization', `Bearer ${token}`);

    expect(res.statusCode).toBe(200);
    expect(res.body).toHaveProperty('totalModules');
    expect(res.body).toHaveProperty('faculties');
    expect(typeof res.body.totalModules).toBe('number');
  });

});

Reading a test

Each test follows the same three-part pattern — sometimes called Arrange, Act, Assert:

  • Arrange — set up what's needed (e.g. beforeAll logs in to get a token)
  • Act — make the HTTP request using Supertest
  • Assert — check the response with expect()

The describe() block groups related tests under a label (the route name). Each it() block is one test case — the string is its human-readable name, which appears in the test output when it passes or fails.

6 Setting Up Playwright

Installation

npm install --save-dev @playwright/test
npx playwright install   # downloads Chromium, Firefox, WebKit

Add playwright.config.ts in the mumodulesapp/ root:

import { defineConfig } from '@playwright/test';

export default defineConfig({
  testDir: './tests/e2e',
  use: {
    baseURL: 'http://localhost:3002',
    headless: true,      // false to watch the browser during development
    screenshot: 'only-on-failure',
  },
});
The app must be running before Playwright tests start. Run npm start in one terminal, then npm run test:e2e in another. Playwright can also be configured to start and stop the server automatically using the webServer option in the config — useful in CI.

Add the test script to package.json:

"test:e2e": "playwright test",

7 E2E Tests (Playwright)

Playwright tests describe user journeys. Each test opens a browser, navigates to a URL, interacts with the page exactly as a user would — clicking, typing, waiting for elements — and then asserts what should be visible.

Login flow — tests/e2e/login.spec.ts

import { test, expect } from '@playwright/test';

test('valid credentials → module list is shown', async ({ page }) => {
  await page.goto('/');

  await page.fill('input[name="username"]', 'testadmin');
  await page.fill('input[name="password"]', 'testpass123');
  await page.click('button[type="submit"]');

  // After login the module list should appear
  await expect(page.locator('.module-list')).toBeVisible();
});

test('wrong password → error message shown', async ({ page }) => {
  await page.goto('/');

  await page.fill('input[name="username"]', 'testadmin');
  await page.fill('input[name="password"]', 'wrongpassword');
  await page.click('button[type="submit"]');

  await expect(page.locator('.error-message')).toBeVisible();
  await expect(page.locator('.module-list')).not.toBeVisible();
});

Module tagging — tests/e2e/tagging.spec.ts

import { test, expect } from '@playwright/test';

// Log in before each test in this file
test.beforeEach(async ({ page }) => {
  await page.goto('/');
  await page.fill('input[name="username"]', 'testadmin');
  await page.fill('input[name="password"]', 'testpass123');
  await page.click('button[type="submit"]');
  await page.waitForSelector('.module-list');
});

test('toggling Data tag and saving persists the change', async ({ page }) => {
  // Search for a specific test module by code
  await page.fill('input[placeholder="Search modules…"]', 'CS101');
  await page.waitForSelector('.module-row');

  // Click the module row to open the edit form
  await page.click('.module-row:first-child');

  // Toggle the Data checkbox
  const dataCheckbox = page.locator('input[name="data"]');
  const before = await dataCheckbox.isChecked();
  await dataCheckbox.click();

  // Save
  await page.click('button:has-text("Save")');

  // Reload and reopen the same module — check the value persisted
  await page.reload();
  await page.fill('input[placeholder="Search modules…"]', 'CS101');
  await page.click('.module-row:first-child');

  await expect(page.locator('input[name="data"]'))
    .toBeChecked({ checked: !before });
});

Department filter — tests/e2e/filters.spec.ts

import { test, expect } from '@playwright/test';

test('selecting a department filters the module list', async ({ page }) => {
  // ... login beforeEach omitted for brevity

  const allModules = await page.locator('.module-row').count();

  // Select "Computer Science" from the department dropdown
  await page.selectOption('select[name="department"]', 'Computer Science');

  const filtered = await page.locator('.module-row').count();
  expect(filtered).toBeLessThan(allModules);

  // Every visible row should belong to CS
  const depts = await page.locator('.module-row .dept-name').allTextContents();
  depts.forEach(d => expect(d).toBe('Computer Science'));
});

Reading a Playwright test

Each test receives a page object — this is the browser tab. The key methods are:

MethodWhat it does
page.goto(url)Navigate to a URL (relative to baseURL)
page.fill(selector, value)Type into an input field
page.click(selector)Click a button or element
page.locator(selector)Find an element (CSS selector or text)
page.waitForSelector(selector)Pause until an element appears in the DOM
page.reload()Reload the page (simulates a browser refresh)
expect(locator).toBeVisible()Assert the element is visible on screen
expect(locator).toBeChecked()Assert a checkbox is (or isn't) checked

8 Running the Tests

API tests only

npm run test:api

E2E tests only

# Terminal 1 — start the app
npm start

# Terminal 2 — run Playwright
npm run test:e2e

Watch mode (API tests during development)

npx jest --watch --testPathPattern=tests/api

Run a single test file

npx jest tests/api/auth.test.js
npx playwright test tests/e2e/login.spec.ts

Playwright with a visible browser (useful for debugging)

npx playwright test --headed

Playwright HTML report

npx playwright show-report

After a Playwright run, an HTML report is generated in playwright-report/ showing which tests passed, failed, and any screenshots captured on failure.

Tip: During development, run API tests in watch mode in one terminal while working on a route. They re-run automatically on file save and give instant feedback. Run Playwright manually when you want to verify the full UI flow.

9 Test Database

Both test layers run against a real MongoDB Atlas database — not a mock. This is intentional: mocking the database hides the kind of real integration bugs that only appear when your code talks to an actual database (wrong field names, missing indexes, type coercion issues).

Setup

  1. Create a separate database in MongoDB Atlas — e.g. MU_test
  2. Seed it with a small set of representative modules and one test user account
  3. Create .env.test in mumodulesapp/ with the test connection string:
MONGODB_URI=mongodb+srv://.../?retryWrites=true
MONGODB_USER_DB_NAME=MU_test
JWT_SECRET=test-secret-not-for-production
JWT_EXPIRATION=1h
PORT=3002

Test data considerations

  • The test database should contain enough modules to make filter tests meaningful (at least 2 departments with multiple modules each)
  • The test user (testadmin) should be seeded into the test database before the suite runs — or created in a beforeAll block
  • Tests that mutate data (e.g. tagging a module) should restore the original state in an afterEach block, so later tests see clean data
  • The production database (MU_transformed_modules) is never touched by any test run
Do not use the production connection string in .env.test. Even a test that only reads data adds load to the production database, and any write test could corrupt live module records.
